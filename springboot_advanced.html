Spring Boot 심화 (Advanced)

게시판 CRUD 만들기에서 벗어날 시간입니다. 실무에서 맞닥뜨리는 성능 최적화, 인증/인가 보안 모델, 트랜잭션의 깊은 이해를 통해 진정한 백엔드 엔지니어로 거듭납니다.

1. AOP (관점 지향 프로그래밍)

모든 컨트롤러나 서비스 메서드에 "실행 시간 측정", "로그 남기기" 같은 코드를 일일이 복붙하시나요? AOP를 사용하면 핵심 비즈니스 로직(Core Concern)과 부가 기능(Cross-cutting Concern)을 완벽하게 분리할 수 있습니다.

메서드 실행 시간 측정 애노테이션 직접 만들기

@Around 어노테이션을 사용하여 메서드 실행 전후를 가로채(Intercept) 시간을 측정합니다.

// 1. 커스텀 어노테이션 생성 @Target(ElementType.METHOD) @Retention(RetentionPolicy.RUNTIME) public @interface LogExecutionTime { } // 2. Aspect(가로채는 녀석) 정의 @Aspect @Component public class ExecutionTimeAspect { // @LogExecutionTime이 붙은 메서드 주변(Around)에서 실행! @Around("@annotation(com.devclub.LogExecutionTime)") public Object logExecutionTime(ProceedingJoinPoint joinPoint) throws Throwable { long start = System.currentTimeMillis(); // 원래 실행되어야 할 핵심 비즈니스 로직 실행 Object proceed = joinPoint.proceed(); long executionTime = System.currentTimeMillis() - start; System.out.println(joinPoint.getSignature() + " 실행 시간: " + executionTime + "ms"); return proceed; } }
💡 AOP의 핵심 용어 Advice(어떤 부가 기능을 할 것인가), Pointcut(어디에 적용할 것인가), JoinPoint(적용되는 구체적인 지점, 메서드). 스프링 내부에 있는 @Transactional 역시 AOP 기술로 만들어진 대표적인 애노테이션입니다.

2. JPA N+1 문제와 Fetch Join 최적화

실무에서 ORM(JPA)을 사용할 때 무조건 겪는 최악의 성능 저하 원인입니다. "사용자 목록 10명을 불렀더니, 각 사용자의 사진을 가져오기 위해 쿼리가 10번 더 나가는 현상"을 N+1 지옥이라고 부릅니다.

@Repository public interface PostRepository extends JpaRepository<Post, Long> { // ❌ 1. 기본 findAll() 사용 시 (N+1 문제 발생) // 게시글(1번)을 가져온 뒤, 반복문을 돌며 댓글(N번)을 따로 조회함 // 🟢 2. Fetch Join으로 해결 (한 방 쿼리) @Query("SELECT p FROM Post p JOIN FETCH p.comments") List<Post> findAllWithComments(); // 🟢 3. @EntityGraph를 이용한 또 다른 해결책 @EntityGraph(attributePaths = {"comments"}) List<Post> findAll(); }
> Hibernate SQL Log (Fetch Join 적용 전)
select post0_.id... from post post0_
select comments0_.post_id... from comment comments0_ where...
select comments0_.post_id... from comment comments0_ where...
WARN: 너무 많은 쿼리가 발생하고 있습니다! (N+1)
> Hibernate SQL Log (Fetch Join 적용 후)
select post0_.id, comment1_.id... from post post0_ inner join comment comment1_ on post0_.id=comment1_.post_id
INFO: 단 한 번의 조인 쿼리로 모두 가져왔습니다.

3. Spring Security와 JWT 인증 모델

과거의 쿠키/세션 방식에서 벗어나, 현대 백엔드 API 서버는 무상태성(Stateless)을 보장하는 JWT (JSON Web Token) 방식을 사용합니다. 스프링 시큐리티의 필터 체인(Filter Chain) 구조를 이해하는 것이 핵심입니다.

SecurityFilterChain 설정 (Spring Boot 3.x 기준)

과거 WebSecurityConfigurerAdapter를 상속받던 방식은 폐기(Deprecated)되었고, 이제는 SecurityFilterChain을 Bean으로 직접 등록합니다.

@Configuration @EnableWebSecurity public class SecurityConfig { private final JwtAuthenticationFilter jwtFilter; @Bean public SecurityFilterChain filterChain(HttpSecurity http) throws Exception { http // REST API이므로 CSRF 보호 비활성화 .csrf(csrf -> csrf.disable()) // 세션을 사용하지 않음 (STATELESS) .sessionManagement(session -> session.sessionCreationPolicy(SessionCreationPolicy.STATELESS)) // URL별 권한 설정 .authorizeHttpRequests(auth -> auth .requestMatchers("/api/auth/**").permitAll() // 로그인, 회원가입은 누구나 접근 가능 .requestMatchers("/api/admin/**").hasRole("ADMIN") // 관리자만 접근 가능 .anyRequest().authenticated() // 나머지는 인증된 사용자만 ) // 우리가 만든 커스텀 JWT 필터를 기본 UsernamePassword 필터 '앞'에 끼워넣기 .addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class); return http.build(); } }

4. @Transactional의 심연 (전파 속성)

데이터베이스 트랜잭션(모두 성공하거나 모두 실패하거나)을 관리하는 @Transactional. 실무에서는 "게시글 작성은 실패해서 롤백되더라도, 로그(Log)는 DB에 무조건 남아야 한다" 같은 복잡한 요구사항이 생깁니다.

Propagation.REQUIRES_NEW

기존 트랜잭션에 참여하지 않고 무조건 독립적인 새로운 트랜잭션을 여는 전파 속성입니다. 부모 트랜잭션이 롤백되어도, 얘는 성공하면 커밋됩니다.

@Service public class OrderService { private final LogService logService; @Transactional // 부모 트랜잭션 시작 public void processOrder(Order order) { try { // 주문 처리 로직... if (order.isInvalid()) throw new RuntimeException("재고 부족!"); } catch (Exception e) { // 에러가 나서 롤백되더라도, 이 로그 기록은 DB에 남아야 함! logService.saveFailLog(order.getId(), e.getMessage()); throw e; } } } @Service public class LogService { // 기존 트랜잭션을 잠시 멈추고 새로운 트랜잭션을 파서 실행함 @Transactional(propagation = Propagation.REQUIRES_NEW) public void saveFailLog(Long orderId, String msg) { logRepository.save(new Log(orderId, msg)); } }
> Transaction Execution Flow
1. [Transaction-1] OrderService 시작
2. [Exception] RuntimeException: 재고 부족!
3. [Transaction-1] Suspend (일시 중지)
4. [Transaction-2] LogService 시작 (REQUIRES_NEW)
5. [Transaction-2] Commit (로그 저장 완료)
6. [Transaction-1] Resume & Rollback (주문 취소 처리)

5. 글로벌 예외 처리 (@RestControllerAdvice)

각 컨트롤러마다 `try-catch`로 도배를 하면 코드가 더러워집니다. 스프링은 애플리케이션 전역에서 발생하는 예외를 한 곳에서 낚아채서, 프론트엔드에게 일관된 JSON 형식으로 에러를 응답하는 우아한 방법을 제공합니다.

// 모든 @RestController에서 발생하는 에러를 감지하는 어노테이션 @RestControllerAdvice public class GlobalExceptionHandler { // CustomException 이라는 내가 만든 에러가 터졌을 때 실행됨 @ExceptionHandler(CustomException.class) protected ResponseEntity<ErrorResponse> handleCustomException(CustomException e) { // 공통 응답 객체(ErrorResponse)를 만들어 일관된 에러 JSON 반환 ErrorResponse response = new ErrorResponse(e.getErrorCode(), e.getMessage()); return new ResponseEntity<>(response, e.getHttpStatus()); } // 파라미터 유효성 검사(@Valid) 실패 시 발생하는 에러 잡아내기 @ExceptionHandler(MethodArgumentNotValidException.class) protected ResponseEntity<ErrorResponse> handleValidationException(MethodArgumentNotValidException e) { String msg = e.getBindingResult().getAllErrors().get(0).getDefaultMessage(); return new ResponseEntity<>(new ErrorResponse("BAD_REQUEST", msg), HttpStatus.BAD_REQUEST); } }
💡 일관된 응답의 중요성 프론트엔드(React, iOS) 개발자가 에러 처리를 쉽게 하려면, 서버가 보내는 에러 JSON의 형태가 항상 똑같아야 합니다. @RestControllerAdvice는 이 규격을 강제하는 가장 좋은 패턴입니다.
