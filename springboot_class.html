1. 스프링 부트의 마법

과거에는 자바로 서버를 띄우려면 수십 장의 설정 파일을 써야 했습니다. 하지만 스프링 부트는 단 하나의 어노테이션(@)과 한 줄의 코드로 내장된 톰캣(Tomcat) 서버를 띄워줍니다.

package com.devclub.app; import org.springframework.boot.SpringApplication; import org.springframework.boot.autoconfigure.SpringBootApplication; // 이 마법의 골뱅이(@) 하나가 모든 복잡한 설정을 대신해 줍니다. @SpringBootApplication public class DemoApplication { public static void main(String[] args) { // 서버 실행! SpringApplication.run(DemoApplication.class, args); } }

2. 컨트롤러 (Controller)와 API

사용자가 웹 브라우저나 앱에서 무언가를 요청하면, 가장 먼저 마중을 나가는 안내 데스크 직원이 바로 컨트롤러(Controller)입니다.

@RestController 와 @GetMapping

어떤 주소(URL)로 들어왔을 때 어떤 대답을 해줄지 정해주는 과정입니다.

// 이 클래스는 API 요청을 받는 컨트롤러입니다! 라고 선언 @RestController public class HelloController { // 누군가 "/hello" 주소로 GET 요청을 보내면 이 메서드를 실행해라! @GetMapping("/hello") public String sayHello() { return "안녕하세요, 스프링 부트입니다! 🌱"; } }
> Browser - http://localhost:8080/hello
안녕하세요, 스프링 부트입니다! 🌱

3. 자바 객체를 JSON으로 자동 변환

현대 백엔드 개발의 핵심은 프론트엔드(React 등)에게 JSON 데이터를 던져주는 것입니다. 스프링 부트는 우리가 자바 객체(클래스)를 그대로 던지면, 알아서 예쁜 JSON으로 변환해 줍니다.

@RestController public class UserController { // 간단한 유저 클래스 (보통은 별도 파일로 분리합니다) class User { public int id; public String name; public User(int id, String name) { this.id = id; this.name = name; } } // "/api/user" 로 요청하면 User '객체'를 반환합니다. @GetMapping("/api/user") public User getUserInfo() { User user = new User(1, "김스프링"); return user; // 객체를 리턴! } }
> API Response (GET /api/user)
{
  "id": 1,
  "name": "김스프링"
}
💡 Jackson 라이브러리의 힘 Node.js에서는 res.json()을 써야 했지만, 스프링 부트 안에는 'Jackson'이라는 라이브러리가 내장되어 있어 자바 객체를 알아서 JSON 문자열로 바꿔서 프론트엔드에 보내줍니다.

4. 핵심 철학 : 의존성 주입 (DI) ★가장 중요

스프링을 스프링답게 만드는 가장 중요한 개념입니다. 개발자가 직접 new 키워드로 객체를 만들지 않고, 스프링 컨테이너가 알아서 객체를 만들고 조립(주입)해 주는 마법입니다.

@Service 와 생성자 주입

비즈니스 로직(실제 일처리)을 담당하는 Service를 만들고, Controller가 이를 사용할 때 스프링이 알아서 가져다줍니다.

// 1. 서비스 클래스 (실제 일꾼) // @Service를 붙이면 "스프링아, 이 객체는 네가 관리해!" 라는 뜻입니다. (IoC 제어의 역전) @Service public class UserService { public String findUserName() { return "개발동아리 부원"; } } // 2. 컨트롤러 (안내 데스크) @RestController public class UserController { private final UserService userService; // Controller가 태어날 때, 스프링이 알아서 UserService 객체를 찾아서 넣어줍니다! (의존성 주입) // 개발자가 "userService = new UserService();" 를 할 필요가 없습니다. public UserController(UserService userService) { this.userService = userService; } @GetMapping("/api/name") public String getName() { return userService.findUserName(); } }
💡 왜 이렇게 복잡하게 하나요? 나중에 서비스가 거대해져서 수백 개의 객체가 얽히고설켰을 때, 개발자가 일일이 new로 관리하면 코드가 스파게티가 됩니다. 이 모든 관계 관리를 스프링이라는 거대한 컨테이너가 대신 맡아주어 유지보수를 엄청나게 쉽게 만들어줍니다.
