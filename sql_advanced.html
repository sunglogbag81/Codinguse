SQL 심화 (Advanced)

단순한 쿼리를 넘어 대용량 데이터 처리와 성능 최적화, 그리고 복잡한 통계 분석을 위한 윈도우 함수와 트랜잭션의 세계로 안내합니다.

1. 복잡함을 깨부수는 CTE (WITH 구문)

쿼리 안에 쿼리가 들어가는 서브쿼리(Subquery)가 중첩되면 코드가 스파게티처럼 변합니다. CTE(Common Table Expression)를 사용하면 가상의 임시 테이블을 위로 빼서 코드를 획기적으로 깔끔하게 만들 수 있습니다.

-- 'VIP_Users'라는 가상의 임시 테이블을 먼저 정의합니다. WITH VIP_Users AS ( SELECT user_id, SUM(amount) AS total_spent FROM orders GROUP BY user_id HAVING SUM(amount) > 1000000 ) -- 위에서 만든 가상 테이블을 진짜 테이블처럼 가져다 씁니다. SELECT u.name, v.total_spent FROM users u JOIN VIP_Users v ON u.id = v.user_id;
💡 가독성과 재사용성의 끝판왕 CTE를 쓰면 로직이 '위에서 아래로' 흐르기 때문에 나중에 코드를 유지보수하는 동료(혹은 미래의 나)가 읽기 매우 편해집니다. 여러 개의 WITH 절을 콤마(,)로 연결해서 쓸 수도 있습니다.

2. 분석의 꽃: 윈도우 함수 (Window Function)

"부서별로 연봉 순위를 매겨줘", "이전 달 대비 매출 증감률을 구해줘" 같은 요구사항은 기존 GROUP BY로는 원본 데이터를 잃어버려서 처리하기 힘듭니다. 원본 행(Row)을 유지하면서 집계와 순위를 계산하는 것이 윈도우 함수입니다.

RANK() 와 PARTITION BY

SELECT name, department, salary, -- 부서(department)별로 파티션을 나누고, -- 연봉(salary) 내림차순으로 순위(RANK)를 매겨라! RANK() OVER ( PARTITION BY department ORDER BY salary DESC ) AS dept_rank FROM employees;

3. 트랜잭션 (Transaction)과 락 (Lock)

계좌 이체를 할 때 '내 통장에서 돈이 빠져나가는 것'과 '상대방 통장에 돈이 들어가는 것'은 반드시 운명을 함께해야 합니다. 둘 중 하나라도 실패하면 전부 없던 일(Rollback)로 되돌리는 것이 트랜잭션입니다.

ACID 원칙과 비관적 락 (FOR UPDATE)

수강신청이나 티켓 예매처럼 여러 명이 동시에 같은 데이터를 수정하려고 할 때, 데이터가 꼬이는 것을 막기 위해 자물쇠(Lock)를 겁니다.

-- 트랜잭션의 시작 (이제부터 하는 작업은 한 묶음이다!) START TRANSACTION; -- FOR UPDATE: 내가 이 데이터를 업데이트 할 거니까, -- 내 트랜잭션이 끝날 때까지 다른 사람들은 이 데이터 읽지도, 건드리지도 마! (비관적 락) SELECT balance FROM accounts WHERE id = 1 FOR UPDATE; -- 데이터 수정 로직 진행 UPDATE accounts SET balance = balance - 50000 WHERE id = 1; UPDATE accounts SET balance = balance + 50000 WHERE id = 2; -- 모든 작업이 완벽하게 성공했으므로 DB에 영구 반영! -- (만약 중간에 에러가 났다면 ROLLBACK; 을 실행) COMMIT;

4. 성능 튜닝: 인덱스(Index)와 EXPLAIN

수천만 건의 데이터가 있는 테이블에서 WHERE name = '홍길동'을 찾으면 DB는 1번부터 끝까지 다 뒤져봅니다(Full Table Scan). 사전의 '찾아보기(색인)' 역할을 하는 인덱스를 만들면 검색 속도가 수백 배 빨라집니다.

-- 1. 이메일 컬럼에 인덱스(색인) 생성 CREATE INDEX idx_user_email ON users(email); -- 2. EXPLAIN: DB가 이 쿼리를 어떻게 실행할지 계획표를 보여줌 EXPLAIN SELECT * FROM users WHERE email = 'dev@club.com';
> MySQL EXPLAIN Output
typekeyrows
refidx_user_email1
* type이 'ALL'이면 최악(전체 스캔). 'ref'나 'const'가 나오면 인덱스를 잘 탔다는 뜻입니다!
⚠️ 인덱스의 치명적인 단점 인덱스는 조회가 빨라지는 대신, 데이터를 추가(INSERT)하거나 수정(UPDATE)할 때마다 색인 책도 같이 업데이트해야 하므로 쓰기 속도가 느려집니다. 따라서 검색이 자주 일어나는 컬럼에만 신중하게 걸어야 합니다.

5. 집합 연산자와 고급 JOIN

교집합, 합집합 같은 집합의 개념을 DB에 적용합니다. 특히 서로 다른 두 개의 쿼리 결과를 위아래로 이어 붙일 때 UNION을 사용합니다.

-- 1. UNION ALL (합집합 - 중복 포함 위아래로 이어붙이기) -- (참고: 그냥 UNION을 쓰면 중복을 제거하느라 성능이 훨씬 느려집니다) SELECT name, '직원' AS role FROM employees UNION ALL SELECT name, '고객' AS role FROM customers; -- 2. LEFT OUTER JOIN (기준 테이블 살리기) -- INNER JOIN과 달리, 주문 내역이 없는(NULL) 유저도 결과에 일단 포함시킵니다. SELECT u.name, o.order_date FROM users u LEFT JOIN orders o ON u.id = o.user_id;
